Severity Filter — 3 Approaches

The current horizontal chip row overflows on small screens. Here are 3 alternatives — all handle 4 fixed severity levels with counts.

A

2×2 Grid of Chips

Wrap the 4 chips into a 2-column, 2-row grid. Each chip stretches to fill half the screen width. Always visible, never overflows. Counts stay inline.

Pros

  • All options always visible
  • Tappable on any screen size
  • Minimal layout change from current
  • Naturally supports multi-select

Cons

  • Takes more vertical space (~2 rows)
  • Might feel heavy above the list
B

Segmented Control (Equal-Width Buttons)

A single row of 4 equal-width segments that always fills the screen. Labels shorten to fit: just the colored dot + count, with a short label below or abbreviated (C / H / M / L). Works like iOS segmented controls.

Pros

  • Single row — compact vertical footprint
  • Always fits, no overflow possible
  • Familiar native pattern

Cons

  • Labels must be abbreviated
  • Segmented controls are traditionally single-select
  • Smaller tap targets than chips
C

Responsive Chips with flexWrap

Keep the current chip design but set flexWrap: 'wrap' on the container. On wide screens all 4 fit in one row. On narrow screens or with large fonts, chips wrap to a second row naturally. No abbreviation needed.

Pros

  • Smallest code change — just add flexWrap
  • Full labels always shown
  • Graceful on all screen sizes
  • Multi-select friendly

Cons

  • Row count varies by device — less predictable layout
  • Could feel less "designed" than a grid

💡 Recommendation: Option A (2×2 Grid)

It gives the most consistent, predictable layout across all screen sizes while keeping full labels and counts visible. The extra vertical space is minimal (one more row of chips), and for a security app where quickly scanning severity levels matters, having all 4 equally prominent is a plus. It also naturally supports multi-select if you want it later.
