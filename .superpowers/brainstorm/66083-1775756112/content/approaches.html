Map-Centric Redesign: Architecture Approaches

How to structure the single-screen map app — choose the approach that feels right

A

Clean Slate — Rewrite from Scratch

Delete everything. New App.tsx that renders a single MapScreen. Rebuild only what's needed: MapBox view, markers, bottom sheet, WebSocket hook, position hook. Estimated ~8-10 new files total.

Pros

  • Zero legacy baggage — no dead imports, no unused types
  • Smallest possible bundle size
  • Easiest to reason about and maintain
  • Clean dependency tree from day one

Cons

  • Loses reusable code (markers, hooks, WebSocket, API layer)
  • Must rewrite auth flow, axios interceptors, mock mode
  • More upfront work even though result is simpler

Best for: When you want a truly fresh start and are OK rewriting the plumbing

B

Surgical Reduction — Keep Core, Delete the Rest

Keep the working infrastructure (auth, API layer, WebSocket, React Query hooks, MapBox markers, mock data) and delete everything else. Collapse navigation to a single screen. Remove ~80% of screens/components, keep ~20% of proven code.

Pros

  • Proven code stays — markers, hooks, WebSocket already work
  • Auth, token refresh, mock mode all preserved
  • Faster to ship — deletion is faster than rewriting
  • API contract with backend unchanged

Cons

  • May leave some dead types/code that needs cleanup
  • Existing code has patterns designed for multi-screen app
  • Redux/Zustand/React Query triple-layer may be overkill for single screen

Best for: Getting to a working simplified app fastest while keeping battle-tested code

C

Hybrid — New Screen, Existing Infrastructure

Write one new MapScreen.tsx that composes existing hooks and marker components. Replace the entire navigation tree with this single screen. Delete all other screens but keep the hooks/services/API layer intact as a clean foundation.

Pros

  • Fresh UI code, proven data layer
  • New screen is purpose-built for map-only UX
  • Hooks and API layer are already clean and well-tested
  • Natural separation: new presentation, existing plumbing

Cons

  • Some existing components (markers) may need minor adaptation
  • Still need to clean up unused store slices and types

Best for: Clean UI without reinventing the data layer. Recommended approach.

💡 Recommendation: Option C (Hybrid)

Your data layer is solid — usePositions, useDevices, WebSocket integration, the marker components, auth/token management. These are the parts that talk to your backend and ESP32 units. No reason to rewrite them.

What needs to change is the presentation — collapsing 6 tabs and 43 screens into one map view with bottom sheets. That's a new screen composed from existing building blocks.

What gets kept: API layer, React Query hooks, WebSocket service, auth flow, MapBox markers, mock data/mode, theme system

What gets deleted: All navigation stacks, all screens except new MapScreen, AI service, LLM pipeline, analytics, reports, replay/timeline, most store slices

What gets written new: MapScreen.tsx, DeviceBottomSheet, DetectionBottomSheet, StatusChip, SettingsOverlay
