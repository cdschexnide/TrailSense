Reports Feature — Two Architectural Approaches

How should the Reports screen be structured?

A

Hub & Spoke (Recommended)

ReportsScreen as a hub with three sections: Templates, Saved Reports, and Intelligence Brief. Each flows into a detail/generation screen.

Pros

  • Clean entry point — everything discoverable from one screen
  • Matches iOS settings-style navigation users already know
  • Each pillar (export, saved views, briefs) gets equal weight
  • Easy to extend with new template types later

Cons

  • More screens to build (hub + detail screens)
  • Extra tap to get to any specific report
B

Tabbed Single Screen

ReportsScreen with 3 tabs (like the Dashboard's Overview/Signals/Patterns). Templates tab, Saved tab, Brief tab — all inline.

Pros

  • Consistent with DashboardScreen's tab pattern
  • Fewer navigation screens to build
  • Quick switching between pillars

Cons

  • Tabs get crowded when each pillar has its own content
  • Report generation/preview needs a modal or push anyway
  • Saved reports list + template picker + brief all compete for space

Recommendation: Option A

Hub & Spoke keeps the Reports screen clean and scalable. The Dashboard already uses tabs for its analytics views — making Reports a hub differentiates it as a distinct feature rather than "more tabs." The extra navigation depth is justified because report generation involves configuration steps (picking template, filters, time range) that benefit from dedicated screens.
